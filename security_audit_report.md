# RWA Compliance Gateway 安全审计报告

审计更新日期：2026-06-10
审计范围：前端、Go API、SQLite 账本、Solidity 合约及部署脚本

## 执行摘要

本轮审计已修复原项目中影响 MVP 自洽性的主要高风险问题：

- 客户端可影响操作人和角色
- 页面保护与后端授权不同步
- JavaScript 浮点金额与链上精度不一致
- 账本操作缺少原子性和重放保护
- 合规状态未按 Token 隔离
- 未授权账户可触达关键合约状态写入
- 审计记录无法证明未被修改
- Demo 行为与真实外部服务/链上交易边界不清

当前结论：适合本地 Showcase 和进一步安全验证，不适合直接处理真实资产。

## 已完成整改

### 认证与路由

| 控制项 | 当前实现 |
|---|---|
| 公共注册提权 | 服务端强制 `investor` |
| 操作人伪造 | 从 JWT address claim 推导 |
| 特权 API | `RequireRoles` 服务端校验 |
| 钱包错配 | KYC/资产页面要求钱包与登录账户一致 |
| 请求速率 | 按 IP 的分钟窗口限流 |
| CORS | 由 `ALLOWED_ORIGIN` 限制 |

前端角色显示只用于用户体验，不能替代后端授权。

### 金额与账本

| 控制项 | 当前实现 |
|---|---|
| 浮点精度 | API 仅接受最小单位整数字符串 |
| 浏览器大数 | 使用原生 `BigInt` |
| 上限 | 后端拒绝超过 `uint256` 的值 |
| 零值/负数 | 写操作拒绝 |
| 余额不足 | 原子减法检查 |
| 超发 | 申购后总余额不得超过供应量 |
| 重放 | Nonce 唯一约束 |
| 部分提交 | 余额、Nonce、审计写入同一事务 |

### 审计证据

每条资产事件包含：

- 操作人、动作和资产 ID
- 最小单位金额和目标地址
- 请求 Nonce
- 前一个事件哈希
- 当前 SHA-256 事件哈希
- 时间戳和说明

`GET /v1/assets/audit-trail` 返回 `integrityVerified`。修改历史事件会导致
哈希链校验失败。该设计提供篡改检测，不等同于不可篡改存储；生产环境仍需
WORM/SIEM 或链上定期锚定。

### 智能合约

| 控制项 | 当前实现 |
|---|---|
| KYC/黑名单/角色写入 | Admin/Regulator 权限 |
| 资产注册和估值 | Issuer 权限 |
| 白名单 | Issuer/Regulator 权限 |
| 持有人状态 hook | 仅已注册 Token |
| 合规检查 | 绑定具体 Token |
| 账户持仓上限 | 转账前检查 |
| 持有人数量 | Token hook 更新 |
| 冻结 | Token pause + asset status |
| 精度 | 2 位 decimals，测试使用 `parseUnits` |

## 当前残余风险

### High：身份绑定仍非 SIWE

当前钱包地址在注册时提交，前端仅检查已连接钱包是否匹配。攻击者若能控制账户密码，
仍可能使用该账户会话。生产必须使用一次性 challenge、钱包签名、域名和过期时间校验。

### High：Demo KYC/制裁数据不可用于真实业务

`KYC_MODE=demo` 生成确定性验证 ID；`SANCTIONS_MODE=demo` 使用内存名单。两者仅用于
Showcase。生产必须接入持牌供应商、处理 Webhook 签名、重试、版本和审查状态。

### High：尚未完成独立合约审计和标准认证

当前实现为 ERC-3643-aligned MVP，不是完整 T-REX 套件，也没有第三方审计背书。
真实发行前必须完成标准互操作、形式化/模糊测试和独立审计。

### Medium：单实例内存控制

限流和手动制裁名单存储在进程内，不适用于多实例部署。生产应迁移到 API Gateway、
Redis 或持久化合规服务。

### Medium：JWT 和 Relayer 密钥管理

环境变量适合本地开发。生产应使用 KMS/HSM，执行轮换、最小权限、双人审批和余额告警。

### Medium：SQLite 和审计留存

SQLite 适合本地演示，不提供生产级高可用、访问审计和灾难恢复。生产应使用受管数据库、
备份、加密、保留策略和独立审计导出。

### Low：前端会话存储

JWT 当前保存在 `localStorage`，存在 XSS 暴露面。生产建议使用受保护的短期会话、
严格 CSP、依赖治理和 HttpOnly/SameSite Cookie 策略。

## 上线门禁

真实资产上线前必须满足：

1. SIWE 登录及会话吊销。
2. 持牌 KYC/AML/制裁数据源。
3. 特权角色双人复核和链上治理。
4. HSM/KMS 密钥托管与轮换。
5. 独立合约审计、应用渗透测试和依赖扫描。
6. ERC-3643 标准互操作验证。
7. 生产数据库、备份、恢复演练和 SIEM。
8. 法律权属、托管、赎回和破产隔离流程评审。
